Élasticité d'une solution cloud

Types of deployment: 
- Online prediction : tu envoies une requête, tu attends une réponse immédiate. Le modèle tourne à la demande, en temps réel. Latence faible, mais coût par requête plus élevé et débit limité. Cas typique : une API de recommandation qui répond pendant qu'un utilisateur charge une page.
- Batch prediction : tu accumules un volume de données, tu lances l'inférence en une fois sur l'ensemble, tu stockes les résultats. Pas de contrainte de latence, tu optimises pour le throughput. Cas typique : générer les catégories de toutes tes transactions du mois en une seule exécution la nuit.

## REST API
HTTP est un protocole de communication — les règles qui définissent comment deux machines échangent des messages sur internet.
REST est un style d'architecture qui s'appuie sur HTTP pour concevoir des APIs. Toute API REST utilise HTTP, mais HTTP ne se limite pas à REST.
Une requête HTTP contient toujours : une méthode (GET, POST, PUT, DELETE), une URL, des headers (contexte : auth, format). Elle contient un body (la donnée) uniquement sur POST et PUT.
La réponse retourne un code : 2xx (succès), 3xx (redirection), 4xx (erreur côté client), 5xx (erreur côté serveur).
Contenu d'une API — Un endpoint (URL + method), des paramètres (dans l'URL ou le body), un système d'auth (token dans le header), des réponses structurées en JSON avec un code HTTP, et une documentation (souvent OpenAPI/Swagger).
REST API vs API classique — Un REST API est une API qui suit des conventions précises : les URLs sont des noms hiérarchiques (la méthode porte l'action), le serveur est stateless (pas de mémoire entre requêtes, le token est dans chaque header), et les réponses sont en JSON. Ces conventions rendent l'API prévisible — tu peux deviner son comportement sans l'avoir vue.

FastAPI est un framework Python pour construire des APIs — REST APIs en particulier.
Concrètement, il te permet d'écrire le serveur qui reçoit les requêtes HTTP, les traite, et renvoie des réponses JSON. Tu définis tes endpoints en Python, FastAPI gère le reste (validation des données, documentation automatique, gestion des erreurs).
À chaque fois on fait un décorateur, le protocole, le path

FastAPI génère automatiquement une interface Swagger UI (disponible sur /docs) et ReDoc (sur /redoc) à partir de ton code Python — sans configuration. Tu peux y tester chaque endpoint de manière interactive directement dans le navigateur : envoyer des requêtes, voir les réponses, consulter les schémas de données attendus.
Pour appeler l'API en dehors du navigateur, deux options courantes :
curl — outil en ligne de commande pour envoyer des requêtes HTTP.
```bash
curl -X GET "http://localhost:8000/transactions?month=2026-03" \
     -H "Authorization: Bearer <token>"
```
Makefile — pour ne pas retaper les commandes à la main. Tu définis une fois :
```makefile
get-transactions:
    curl -X GET "http://localhost:8000/transactions?month=2026-03" \
         -H "Authorization: Bearer <token>"
```
Et tu appelles simplement make get-transactions.

Un modèle apprend sur des données transformées — pas sur les données brutes. Il faut donc appliquer exactement le même preprocessing à l'entraînement (train) et à l'inférence (predict), sans quoi le modèle reçoit des données qu'il ne reconnaît pas.
Exemples concrets de preprocessing à reproduire fidèlement :

- Normalisation / standardisation (même moyenne et écart-type que ceux calculés sur le train)
- Encodage des variables catégorielles (même mapping catégorie → entier)
- Tokenisation du texte (même tokenizer, même vocabulaire)
- Gestion des valeurs manquantes (même stratégie d'imputation)

L'erreur classique : recalculer les paramètres de normalisation sur les données de production (ex: la moyenne du mois courant) plutôt que d'utiliser ceux calculés sur le train. Le modèle reçoit alors une distribution différente de ce qu'il a appris — les prédictions se dégradent silencieusement.
La bonne pratique : sérialiser les paramètres de preprocessing avec le modèle (via un Pipeline sklearn par exemple) pour garantir que les deux voyagent ensemble et sont toujours cohérents.
Retraining (périodique, ex: annuel)
  → recalculer le preprocessing sur le nouveau dataset complet
  → ré-entraîner le modèle
  → sauvegarder le nouveau pipeline (preprocessing + modèle)

Inférence (mensuelle)
  → charger le pipeline figé
  → appliquer sans recalculer

## Uvicorn

**Uvicorn** est un serveur web ASGI (_Asynchronous Server Gateway Interface_) utilisé pour faire tourner une application FastAPI en production ou en développement.

- FastAPI est un framework — il ne sert pas lui-même les requêtes HTTP ; il a besoin d'un serveur ASGI pour ça
- Uvicorn est le serveur recommandé et est inclus comme dépendance de FastAPI (`pip install fastapi[standard]` installe Uvicorn automatiquement)
- En dev : `uvicorn main:app --reload` (rechargement automatique à chaque modif)
- En prod : on le couple souvent avec **Gunicorn** comme process manager (`gunicorn -w 4 -k uvicorn.workers.UvicornWorker main:app`)

**Alternative** : Hypercorn (aussi ASGI, compatible HTTP/2)

> ASGI est le successeur de WSGI (Flask, Django classique) — il supporte nativement l'async/await Python.

## Sécuriser FastAPI

### 1. Isolation réseau
Déployer l'API sur un réseau interne (pas d'exposition internet) — défense en profondeur, mais insuffisant seul.

### 2. Authentification & autorisation
FastAPI s'intègre nativement avec **OAuth2** via `fastapi.security` :
```python
from fastapi.security import OAuth2PasswordBearer

oauth2_scheme = OAuth2PasswordBearer(tokenUrl="token")
```

- **OAuth2 + JWT** : standard recommandé (stateless, signé)
- **API Keys** : via header `X-API-Key` pour des usages service-to-service
- **Keycloak / Auth0** : Identity Providers (IdP) externes branchables comme middleware

> "Identity Provider" (IdP), pas "Access Provider" — c'est l'entité qui émet et valide les tokens d'identité.

## Locust

Library Python de **tests de charge** (_load testing_) pour APIs et services web.
```bash
pip install locust
```
```python
from locust import HttpUser, task

class MyUser(HttpUser):
    @task
    def hit_endpoint(self):
        self.client.get("/items/1")
```

- Lance une UI web sur `localhost:8089` pour configurer nb d'users et ramp-up
- Simule des utilisateurs concurrents → mesure latence, throughput, erreurs
- Alternative : **k6**, **Artillery** (JS-based)

## Workflow de déploiement FastAPI

### Étapes dans l'ordre

1. **Dockerfile** — containeriser l'app FastAPI + Uvicorn
2. **Build & test local** — itérer vite avant tout push
   > Débugger en prod (GCP) prend ~4× plus de temps qu'en local → valider un maximum en local
3. **Vérifier les credentials** — IAM GCP, secrets, variables d'env (ne pas hardcoder dans l'image)
4. **Push de l'image** — vers Artifact Registry (ou Container Registry)
5. **Déploiement** — Cloud Run, GKE, ou Compute Engine selon le besoin

## Stockage des modèles ML — bonnes pratiques

### ❌ À éviter
Embarquer le modèle directement dans l'image Docker — l'image grossit, et tout réentraînement force un rebuild + redéploiement.

### Options

**Bucket GCP (solution retenue)**
- Modèle stocké sur GCS, chargé au démarrage de l'API
- Réentraînement via Prefect → écrase le fichier sur le bucket → l'API sert automatiquement le nouveau modèle
- Simple, à la main, pas de dépendance supplémentaire

**MLflow Model Registry**
- Versioning, staging/production, metadata — plus complet
- Requiert de déployer et maintenir un serveur MLflow

**Vertex AI**
- Solution fully-managed GCP (entraînement + registry + serving)
- Nécessite d'avoir structuré la pipeline ML dès le début — pas retrofit-friendly

### Takeaway
> Bucket GCP = solution pragmatique pour un projet existant. Vertex AI = à considérer dès le début d'un nouveau projet ML.
