# Notes — dbt-fundamentals

Bascule vers Rust dans l'écosystème data
  Ruff est plus rapide que Flake8 (et non "Flake8") car codé en Rust (vs Python pour   
  Flake8). Même logique : les outils Python-natifs migrent vers Rust pour la performance mémoire
  et vitesse d'exécution.

  ---
  ## dbt — Produits
  - dbt Core : open source, CLI, moteur Python
  - dbt Fusion : nouveau compilateur dbt écrit en Rust (lancé 2024-2025) — remplace le
  compilateur Python dans dbt Core ; beaucoup plus rapide sur les grands projets
  - dbt Cloud : plateforme hébergée par dbt Labs (scheduler, CI/CD, IDE, observability)

  ---
  ## dbt run — matérialisations possibles des modèles :

  ┌─────────────┬────────┬────────────────────────────────────────────────────────────┐
  │    Type     │ Stocké │                        Comportement                        │
  │             │    ?   │                                                            │
  ├─────────────┼────────┼────────────────────────────────────────────────────────────┤
  │ view        │ Non    │ Requête réexécutée à chaque appel. Donnée toujours         │
  │             │        │ fraîche. Coût compute à la lecture.                        │
  ├─────────────┼────────┼────────────────────────────────────────────────────────────┤
  │ table       │ Oui    │ Reconstruite entièrement à chaque dbt run. Lecture rapide. │
  ├─────────────┼────────┼────────────────────────────────────────────────────────────┤
  │             │        │ Ne charge que les lignes nouvelles/modifiées depuis le     │
  │ incremental │ Oui    │ dernier run. Nécessite une stratégie (unique_key ou filtre │
  │             │        │ timestamp). Idéal pour les grandes tables.                 │
  ├─────────────┼────────┼────────────────────────────────────────────────────────────┤
  │ ephemeral   │ Non    │ Jamais matérialisée en base — compilée en CTE dans les     │
  │             │        │ modèles qui en dépendent.                                  │
  └─────────────┴────────┴────────────────────────────────────────────────────────────┘

  **incremental model** : évite de reconstruire toute la table à chaque run — on filtre sur
  updated_at > last_run ou via unique_key. Trade-off : la donnée peut diverger si la
  logique de filtre est mal écrite ; dbt run --full-refresh force une reconstruction
  complète.

  **snapshot** : historisation de type SCD Type 2 — dbt compare l'état courant d'une table
  source avec le snapshot précédent et ajoute une ligne horodatée pour chaque changement
  (colonnes dbt_valid_from, dbt_valid_to). Permet de reconstituer l'état d'un
  enregistrement à n'importe quelle date.
